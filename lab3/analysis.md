# анализ сгенерированных вариантов

## критерии оценке

сравнение выполнялось по пяти критериям:

- корректность алгоритма относительно условия задачи
- соответствие формату ввода и вывода
- устойчивость к ошибкам данных
- читаемость и поддерживаемость кода
- простота интеграции в итоговый файл решения

## Task 1. Graf10

### что требовалось по условию

- прочитать граф из матрицы смежности
- найти все маршруты из K1 в K2 без зацикливаний
- отсортировать маршруты по длине, затем лексикографически вывести число маршрутов и сами маршруты, либо -1

### сравнение вариантов

- 1-claude.py
сильные стороны: корректный DFS по простым путям, правильная сортировка
риски: жесткие имена input.txt/output.txt, привязка к дефолтным K1/K2

- 1-gemini.py
сильные стороны: корректный backtracking, понятная структура функций
риски: интерактивный ввод K1/K2 через input, что нарушает файловый формат

- 1-perplexity.py
сильные стороны: близкий к условию формат файловой обработки
риски: предполагает наличие K1/K2 в файле без валидации структуры и диапазонов

### вывод

лучшие части взяты из 1-claude.py и 1-perplexity.py:

- dfs и сортировка как в 1-claude.py
- файловый сценарий запуска как в 1-perplexity.py

в итоговом варианте добавлены:

- единый способ чтения данных из FileName1
- поддержка K1/K2 из хвоста файла или из аргументов
- проверка диапазона вершин
- корректная запись в FileName2

## Task 2. CalcTree8

### что требовалось по условию

- построить дерево из префиксной записи
- закодировать операторы значениями от -1 до -6
- преобразовать дерево: если левое или правое поддерево
имеет значение 0, свернуть текущее поддерево в число
- вывести указатель на корень полученного дерева

### сравнение вариантов

- 2-claude.py
сильные стороны: почти полная реализация, аккуратная структура узла
риски: избыточный отладочный вывод и лишние представления дерева

- 2-gemini.py
сильные стороны: базовый парсинг и структура дерева присутствуют
риски: преобразование подменяет только детей нулями
а не сворачивает нужное поддерево по условию

- 2-perplexity.py
сильные стороны: корректная идея рекурсивного parse/eval/transform
риски: неоднозначная обработка токенизации
вывод id(root) вместо самого корня как объекта

### вывод

итоговый файл собран на базе сильных сторон
2-claude.py и 2-perplexity.py:

- строгий рекурсивный разбор префиксного выражения
- единая функция evaluate для всех операций
- точное сворачивание поддерева по условию
- явная обработка ошибок (формат, деление на ноль)

## Task 3. HomeDyn17

### что требовалось по условию

- посчитать число N-разрядных K-ичных чисел
- ведущий ноль запрещен
- в записи не должно быть трех нулей подряд

### сравнение вариантов

- 3-claude.py
сильные стороны: корректная DP-модель, есть валидация ограничений
риски: лишняя вспомогательная логика

- 3-gemini.py
сильные стороны: корректные переходы DP и читаемая формулировка состояний
риски: только интерактивный сценарий, меньше контроля формата запуска

- 3-perplexity.py
сильные стороны: компактная реализация и правильные переходы
риски: инициализация для длины 1 включает случай с нулевым
первым разрядом, что противоречит требованию N-разрядности

### вывод

итоговый вариант реализует DP-идею,
но с корректной инициализацией:

- для длины 1 учитываются только ненулевые первые цифры
- состояния хранят количество чисел,
которые заканчиваются на 0, 1 или 2 нуля
- добавлены проверки ограничений N и K
